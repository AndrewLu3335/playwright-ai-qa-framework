# Lace Up API Test Foundation Design

## Goal

Extend the Playwright AI QA framework with a deterministic Lace Up API test layer that authenticates through the local E2E login endpoint and validates public, unauthorized, and authenticated API behavior.

## Scope

This phase covers API authentication and read-only API tests. It does not add Lace Up UI tests, create run records, query PostgreSQL directly, automate Strava OAuth, or start the Lace Up Docker stack from the Playwright repository.

## Runtime Assumptions

The Lace Up backend is started separately with its local E2E override:

```bash
cd /Users/lujingsheng/lace_up
docker compose -f compose.dev.yaml -f compose.e2e.yaml up -d db backend
```

The Playwright project uses `http://localhost:8000` by default. `LACEUP_API_URL` may override that URL without changing source code.

## Playwright Projects

The existing `chromium` project remains responsible for SauceDemo tests under `tests/ui/`.

Two Lace Up projects are added:

- `laceup-auth`: runs only `tests/laceup/auth.setup.ts` and creates authenticated storage state.
- `laceup-api`: runs only `tests/laceup/api/**/*.spec.ts`, loads the authenticated state, and depends on `laceup-auth`.

Explicit `testMatch` patterns prevent SauceDemo UI tests from running in Lace Up projects and prevent Lace Up tests from running in the SauceDemo browser project.

## Authentication Flow

`auth.setup.ts` sends:

```http
POST /api/test/login/
Content-Type: application/json

{"username":"e2e_playwright_api"}
```

The setup verifies a successful response and saves the resulting Django Session cookie to:

```text
playwright/.auth/laceup.json
```

The directory is already ignored by Git. Tests and reports must not print or attach the session cookie.

`LACEUP_E2E_USERNAME` may override the default username, but the value must begin with `e2e_` because the Lace Up endpoint enforces that prefix.

## Test Coverage

`tests/laceup/api/runs.spec.ts` contains three read-only tests:

1. The public health endpoint returns `200` and `{ "status": "ok" }`.
2. A fresh unauthenticated API context cannot access `/api/runs/` and receives `401` or `403` according to Django REST Framework behavior.
3. The authenticated project context can access `/api/runs/`, receives `200`, and returns a JSON array.

The tests create no run records and therefore require no data cleanup.

## Configuration And Commands

`package.json` changes `test:api` to select the `laceup-api` project:

```text
npx playwright test --project=laceup-api
```

The normal `npm test` command runs the SauceDemo and Lace Up projects. Lace Up must be running before commands that include Lace Up projects.

## Failure Reporting

API tests use the existing framework test export so they remain part of the standard HTML and JSON reports. API requests do not create screenshots or page evidence; Playwright still records assertion errors, response details included by the test, duration, retries, and attachments generated by the runner.

The AI failure analyzer remains optional and consumes the same JSON result file. This phase does not change AI classification or report rendering.

## Verification

Implementation is complete when:

- Playwright lists `laceup-auth` and `laceup-api` without configuration errors;
- the auth setup creates ignored storage state without exposing its cookie;
- all three Lace Up API tests pass against local Docker;
- SauceDemo UI test discovery remains isolated to `chromium`;
- `npm run test:api` runs the authentication dependency and Lace Up API tests;
- TypeScript compilation and `git diff --check` pass;
- existing uncommitted License changes remain untouched.
